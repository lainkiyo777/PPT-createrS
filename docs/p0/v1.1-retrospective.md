# v1.1 蓝色模板 PPT 失败复盘

> 状态：v1.1 判定为失败样例。本次只做证据复盘、模板风格画像和 Skill 门禁修订，不重新生成 PPT，不修改现有 `presentation.pptx`。

## 复盘范围与事实证据

检查对象：

- `projects/wind-blue-template-v1.1/output/`
- `skills/ppt-creaters/references/deck-library/inbox/蓝色模板.pptx`
- 模板审计产物 `.tmp-wind-blue-template/template-inspect/` 和 `template-frame-map.json`

关键事实：

- output 有 24 份 slide spec、24 张 preview、24 张 final，说明文件数量门禁通过，但不代表视觉设计通过。
- 模板有 11 个源页面、7 个媒体资源、16:9 画布，源模板没有原生 chart 或 table 页面。
- `template-frame-map.json` 的 24 个输出页面全部使用 `reuseMode: duplicate-slide`，并写明“edits are limited to inherited text elements”。
- `selected-style.yaml` 的规则是“只替换继承文本框”“不新增覆盖模板内容的独立版式”。这正是本次退化的主要设计约束。

- 生成前没有 `references/deck-library/profiles/蓝色模板/style-profile.yaml`。`selected-style.yaml` 不是 style profile，它只记录了颜色、字体和复用规则。
- `qa-report.md` 的 artifact-tool 结构检查通过，模板计划和模板保真检查通过；这些检查只说明文件结构和源模板保留情况，没有证明叙事、信息密度和主视觉质量通过。

## 1. 哪些页面只是机械套用了模板背景

这不是指 PPTX 技术上完全没有编辑文本；而是指页面设计层面没有重新选择适合内容的构图，主要依赖源页的背景、几何形状和继承文本框。

| 源模板页/家族 | 输出页面 | 机械套用表现 |
| --- | --- | --- |
| 源页 1：封面 | 01、24 | 第 24 页“总括结论”仍回到封面家族；源页 11 的 closing 家族没有被建模，说明结尾只是找了一个能容纳文字的背景。 |
| 源页 3：章节/主张 | 03、08、16 | “数据资产定位”和两个阶段标题使用同一构图，内容没有新增主视觉或章节差异。 |
| 源页 4：卡片事实 | 04、09、17 | 数据集信息、测试集划分、阶段二方法路线被压到同一种卡片容器中，语义不同但阅读方式相同。 |
| 源页 5：时间线/序列 | 05、10、18 | 覆盖时间跨度、预测任务指标、功率模型结果共用序列构图；尤其第 18 页模型对比不是时间线语义。 |
| 源页 6：系统卡片 | 06、11、19、22 | 字段体系、模型路线、MoE 结构、闭环方案均使用同一组模块卡片，没有真实的数据流、输入输出或箭头关系。 |
| 源页 7：密集证据 | 07、12、20 | 数据质量、full test 对比、误差分析都挤入同一 dashboard 壳，页面之间没有不同的证据主次。 |
| 源页 8/10：步骤或总结 | 13、15、21、23 | 消融、阶段总结、闭环衔接、验证计划被套入步骤/总结壳，缺少因果图、决策树或行动优先级。 |

结论：24 页虽然看起来“保持了模板”，但大部分是“复制源页 + 改文本”，不是“从模板规则重新设计页面”。

## 2. 哪些页面信息量被过度压缩

所有 24 份 spec 的 `content_blocks` 都只有 2 项，其中第二项是“正文由模板继承文本框确定性渲染”的流程说明，不是有效叙事内容。实际内容因此被压成“一个结论 + 少量短句”。高风险页面如下：

- **03 数据资产定位**：只有抽象判断，没有字段分组、数据链路或可用性证据。
- **05 机组覆盖与时间跨度**：覆盖范围、机组差异、时间跨度被压成同一条序列，无法看出样本结构。
- **07 数据质量与基础统计**：质量结论、分布、异常和关键统计缺少分区，数字只能作为小卡片阅读。
- **09 阶段一数据口径与测试集划分**：训练/验证/测试关系没有清晰的切分图或时间轴。
- **10 预测任务与评价指标**：预测时距、变量、评价指标和业务含义没有形成一张任务定义图。
- **12 统一 full test 下的模型结果**：模型比较没有完整排序、差值和误差解释，读者只能看到少量指标。
- **13 FuXi-CFD 消融**：特征、历史窗口和结论没有一一对应，消融逻辑被压成短标签。
- **14 OpenOA clean 微调结果**：清洗集、微调关系和收益没有用前后对照呈现。
- **15 阶段一成果总结**：结论、限制、进入阶段二的条件没有分层。
- **18 功率映射模型结果**：有 chart-data，但模板序列卡片仍承担主要视觉，图表与结论层级不够突出。
- **20 误差分布与高风速难点**：分布形态、高风速区间和改进方向没有被拆成图表 + 解释。
- **23 下一步验证计划**：验证对象、环境、验收指标、顺序和责任动作缺少行动结构。

这不是简单“字体加大”即可解决的问题；应先删除次要信息、分拆页面或换 page family，再做排版。

## 3. 哪些页面缺少主视觉或关键图表

模板审计显示源 PPT 没有原生 chart/table 页面，因此必须由 slide spec 和确定性 chart renderer 补齐主视觉。本版没有做到：

- `output/charts/` 为空。
- 只有 **12、18** 页引用了 `data/chart-data/*.json`；其余量化页面的 `chart_data` 为 `none`。
- **05** 缺少覆盖时间轴或机组覆盖图。
- **07** 缺少数据质量分布/异常统计图。
- **09** 缺少时间切分示意图。
- **10** 缺少预测任务与 horizon 指标图。
- **13** 缺少消融对比图。
- **14** 缺少 clean test 前后对照图。
- **20** 缺少误差分布、高风速分桶或残差图。
- **23** 缺少验证路径/验收门槛图。
- **06、11、19、22** 虽然被标为 system/architecture，但只是模板卡片和文字，没有真正的模型结构、两阶段接口或闭环箭头。

以后凡是“比较、趋势、分布、提升、路径、架构”类结论，都必须在 spec 中明确 `dominant_visual` 或 `chart_data`；没有主视觉不得进入 preview gate。

## 4. 哪些页面字体、留白、层级不符合原模板

问题不是模板本身的蓝色不好，而是模板源页的字体和空白是按不同 page family 设计的，本版把它们当成了通用文本容器：

- spec 全局写成 title 48、body 30、chart label 26，未建立封面、章节、卡片、dashboard、focal-result 各自的字号和标题位置。
- 第 01 页为避免“短时功率预测模型”换行，标题被压到 50；第 24 页也复用了封面标题容器。这解决了溢出，却暴露出没有真正的 closing family。
- 第 07、12、18 等数据页把较多数字和说明塞入继承的小文本框；即使形式上达到 body 最小字号，视觉上仍是“标题很大、正文被压扁”。
- 源模板有大面积摄影、蓝色几何和安全留白；内容页没有重新分配这些空白给图表和结论，导致装饰面积与信息面积失衡。
- 页间 title baseline、卡片内部 padding、标题与主视觉的距离随源页变化，但 spec 没有 `layout_principles`、`spacing` 或 family-level spacing 约束。
- `qa-report.md` 只报告了结构 QA 和字号下限，没有检查“标题—主视觉—结论”的层级，也没有检查留白是否服务于叙事。

## 5. 模板是否被解析成 style-profile.yaml

**生成时没有。** 当时只有：

- `selected-style.yaml`：记录 palette、字体和“继承文本框”规则；
- `.tmp-wind-blue-template/template-inspect/`：原始审计、布局 JSON、媒体和预览；
- `template-frame-map.json`：输出页到源页的复制映射。

这些文件没有把模板抽象成可执行的 `color_palette`、`typography`、`composition_language`、`spacing`、`layout_principles`、`page_rhythm`、`chart_style`、`image_treatment`、`prohibited_behaviors`。因此生成器没有机会判断“该内容是否适合这个源页”。本次复盘新增了该 profile，但它不 retroactively 让旧 PPT 通过。

## 6. 当前是否只是把模板当作背景图使用

准确说法是：**技术实现不是单纯把一张背景 PNG 放进 PPT，但设计结果在功能上等价于模板背景/外壳复用。**

证据是：

1. `template-frame-map.json` 对 24 页全部使用 `duplicate-slide`。
2. map 的说明是“edits are limited to inherited text elements”。
3. `selected-style.yaml` 明确禁止新增覆盖模板内容的独立版式。
4. 最终 `presentation.pptx` 每页只放一张 full-canvas final image，源页的几何和图片先被保留，再将文字扁平化。

所以这不是严格意义上的“只插入背景图”，但没有完成模板学习、家族选择和页面重构，仍然属于“模板外壳 + 改字”。

## 7. 为什么生成结果比 v1.0 的科技风弱

对照既有 v1.0 的科技风生产方式，差异在生产策略而非颜色：

- v1.0 的科技风允许每页围绕结论设计独立的图像构图、视觉焦点和信息层级；本版优先保留源页，导致内容反过来迁就文本框。
- v1.0 的视觉系统通过统一背景、光效和模块化信息块建立连续节奏；本版虽然颜色统一，但 24 页重复少数源页家族，节奏单一。
- v1.0 中图表/架构可以成为页面主视觉；本版没有把 chart-data 变成 `charts/` 中的可见资产，数据页因而退化成文字卡片。
- v1.0 的生产流程先确定页面构图，再渲染文字；本版的 `editTargets` 先锁定模板继承文本框，再压缩内容适配，视觉设计空间被提前关闭。
- v1.1 的“模板保真检查通过”只验证了源模板没有被破坏，不能替代“内容是否被模板正确承载”的视觉验收。

结论：本版不是“模板风格不如科技风”，而是把模板保真误当成了模板应用质量，牺牲了叙事、主视觉和信息层级。

## 8. 如何修改 Skill，避免模板套壳

已把下面的流程写入 `skills/ppt-creaters/SKILL.md`、`references/p0-contract.md` 和 `scripts/validate_p0.py`：

1. **模板导入门禁**：检测到 `template_source` 后，先生成并校验 `references/deck-library/profiles/<template-name>/style-profile.yaml`；没有 profile 或缺少任一必需 section，直接 `build_status: failed`。
2. **先画像、后选家族**：profile 必须描述颜色、字体角色、页面家族、布局模式、标题位置、内容密度、图表语言、装饰边界和不适用场景；不能只写 palette 或背景路径。
3. **逐页绑定家族**：每个 slide spec 必须引用 `page_family`，并声明 `dominant_visual` 或 `chart_data`。内容语义与源家族不匹配时，回到 spec 换家族或拆页，不得机械复制源页。
4. **视觉充分性门禁**：比较、趋势、分布、提升和架构类页面没有图表、流程图或主视觉时，不得通过 preview gate；`charts/` 需要有实际渲染资产，而不是只有 JSON 数据。
5. **密度和层级门禁**：使用 profile 的 family-level title position、字号和最大承载量；超出容量时删减、拆页或改布局，不得继续缩小正文。
6. **模板不是背景**：保留品牌色、几何和摄影资产，但允许在模板安全区内为内容建立新的图表、箭头、数据块和主视觉；禁止“只替换继承文本框”作为通用生产策略。
7. **重复率检查**：同一源家族只能在语义相同且构图有明确变化时复用；封面不能自动充当 closing，时间线不能自动充当模型比较，卡片不能自动充当架构图。
8. **最终 QA 分层**：模板保真、文本溢出、数据一致性只是必要条件；必须新增“主视觉是否存在、图表是否能读、留白是否服务叙事、家族是否误用”的视觉检查。

## 新增的硬规则

模板导入后必须先生成：

```text
references/deck-library/profiles/<template-name>/style-profile.yaml
```

`style-profile.yaml` 至少包含：

- `color_palette`
- `typography`
- `composition_language`
- `spacing`
- `layout_principles`
- `page_rhythm`
- `chart_style`
- `image_treatment`
- `prohibited_behaviors`

没有该文件或缺少这些 section 时，不允许使用该模板生成整套 PPT；必须停止在模板画像门禁，不得通过修改 PPT 合成代码绕过。

## 9. Template strategy correction

本次失败不是因为允许修改模板过多，而是因为错误地将模板定义为固定版式库。正确目标是“风格学习 + 新内容重新构图”，而不是“复制源页 + 替换文字”。

The corrected contract has three modes:

- `style-reference` is the default. It inherits color, font language, whitespace, geometry, icon, image-treatment, chart language, and rhythm. Each page is recomposed from semantics and `source_slide_reference` may be empty.
- `adaptive-layout` may reference page families and broad composition, while allowing text containers and visual elements to move, resize, be deleted, or be added. It may add charts, nodes, arrows, and architecture relationships.
- `strict-template` is available only after an explicit user choice. Uploading a PPTX never confirms it.

`style-profile.yaml` now describes transferable visual language rather than making coordinates or duplicate-slide mappings the core. The image-first flow is independent slide specs -> dominant visual -> style-reference prompt -> preview review -> final images -> assembly. Exact Chinese, key numbers, tables, and charts remain deterministic renders.

This round changes only the design contract, validator, tests, and three-page single-slide design contract; no complete PPT was regenerated.
